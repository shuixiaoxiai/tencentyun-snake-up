# Web 化设计文档

## 背景与目标

当前项目是命令行抢购脚本，后续需要 Web 化。

目标：

- 用户可以通过页面开始抢购。
- 用户可以在页面内扫码登录腾讯云。
- 登录成功后自动读取 cookie。
- 后台启动抢购任务并显示下一场倒计时。
- 抢购完成后页面显示成功或失败结果。
- 用户可以通过唯一 id 查询已有任务状态。

## 用户流程

### 开始抢购

1. 用户输入唯一 id，推荐使用邮箱。
2. 用户点击“开始抢购”。
3. 页面显示腾讯云登录二维码。
4. 用户扫码登录。
5. 后端读取并保存 cookie。
6. 后台任务进入下一场抢购倒计时。
7. 到点后自动抢购。
8. 页面显示抢购结果。

### 查询任务

1. 用户输入唯一 id。
2. 用户点击“查询”。
3. 如果任务存在，页面显示任务状态、下一场倒计时或抢购结果。
4. 如果任务不存在，页面提示未找到任务。

## 核心需求

- 用户 id 必须唯一，推荐使用邮箱。
- 开始抢购时，如果 id 已存在，应提示重复。
- 查询必须基于已有 id，不自动创建任务。
- 支持页面内显示二维码。
- 支持多用户任务并发。
- 每个任务独立保存 cookie、状态和结果。
- 页面不展示 cookie、token 等敏感信息。

## 技术设计

- 后端使用 FastAPI。
- 前端使用原生 HTML、CSS、JavaScript。
- Playwright 负责打开腾讯云登录页并获取二维码。
- 抢购逻辑从 `snap_up_server.py` 拆成后台任务。
- 任务数据使用本地文件存储，统一放在 `data/` 目录。
- 每个任务使用独立 session、cookies、headers 和状态文件。

## 状态设计

任务状态包括：

- `created`：任务已创建。
- `waiting_login`：等待用户扫码登录。
- `logged_in`：登录成功，cookie 已读取。
- `waiting_seckill`：等待下一场抢购。
- `running`：正在执行抢购。
- `success`：抢购成功。
- `failed`：抢购失败。

## 接口设计

### 创建任务

```http
POST /api/tasks
```

用途：创建抢购任务并启动二维码登录流程。

请求参数：

```json
{
  "user_id": "user@example.com"
}
```

### 查询任务

```http
GET /api/tasks/{user_id}
```

用途：查询任务状态、下一场时间、倒计时和抢购结果。

### 获取二维码

```http
GET /api/tasks/{user_id}/qr
```

用途：获取当前任务的腾讯云登录二维码图片。

### 订阅任务事件

```http
GET /api/tasks/{user_id}/events
```

用途：推送登录状态、倒计时变化和抢购结果。

## 数据设计

任务数据按用户 id hash 后分目录保存：

```text
data/tasks/<user_id_hash>/
├── state.json
├── cookies.json
└── result.json
```

文件说明：

- `state.json`：保存用户 id、任务状态、下一场时间、倒计时等信息。
- `cookies.json`：保存该任务独立的腾讯云 cookie。
- `result.json`：保存抢购接口返回结果。

`data/` 目录应加入 `.gitignore`，避免提交敏感数据。

## 抢购策略

- 每天北京时间有两场：`10:00` 和 `15:00`。
- 登录成功后，任务等待下一场抢购。
- 秒杀前 5 分钟重新读取 cookie。
- 根据 cookie 中的 `skey` 计算 `x-csrf-token`。
- 到点后并发抢购地域 `[1, 4, 8]`。
- 抢购完成后保存接口返回，并更新任务状态。

## 约束与暂不实现

- v1 只考虑本地部署。
- 后续可通过内网穿透暴露公网。
- v1 暂不做访问密码。
- v1 暂不引入数据库。
- v1 暂不使用前端框架。
- v1 不支持用户修改已存在任务，只支持查询已有任务。
